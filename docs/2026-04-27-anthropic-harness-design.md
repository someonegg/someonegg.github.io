# Anthropic Harness Design 笔记

文章链接：<https://www.anthropic.com/engineering/harness-design-long-running-apps>  
发布日期：**2026 年 3 月 24 日**  
来源：Anthropic Engineering Blog

## 一句话总结

这篇文章最值得记住的不是某个三 Agent 配方，而是一个更底层的判断：**长时程 Agent 的上限，越来越取决于 harness 设计，而不是单轮模型能力。**

---

## 一、这篇文章在解决什么问题

作者处理的是两个表面不同、底层相通的问题：

- 为什么模型做前端设计时，常常“能用但平庸”
- 为什么模型做长时程应用开发时，常常中途跑偏、草草收尾，或者留下半成品

文章的答案不是继续堆 prompt，而是把生成、评审、规划、验收、上下文管理这些机制单独设计出来。

换句话说，问题不再只是“模型会不会做”，而是：

- 谁负责生成
- 谁负责否决
- 什么算完成
- 长任务跨多小时后如何不失真

---

## 二、作者抓到的两个核心失败模式

### 1. 长任务会逐渐失去连贯性

文章认为，长时任务的一个典型问题是模型随着上下文累积而逐渐失焦。  
有些模型还会在接近上下文上限时提前收尾，作者把这种现象称为 `context anxiety`。

这里最重要的观点不是“上下文会满”，而是：

**上下文窗口不是稳定工作区，而是一个会逐渐恶化的认知环境。**

因此，长任务设计的重点不是一味延长会话，而是主动管理上下文，包括：

- 什么时候压缩
- 什么时候重开新 agent
- 交接时保留什么结构化状态

### 2. 生成者几乎不会严格否定自己

文章把另一个关键问题归结为自评失真：

- 设计任务里，生成者往往会高估自己的审美质量
- 编程任务里，生成者也会低估 bug、误判完成度、放过“看起来差不多”的功能

所以作者把一个很重要的工程原则说清楚了：

**不要让负责产出的人同时定义产出质量。**

这其实很像真实软件团队里的角色分离：

- 开发不等于验收
- 做出来不等于可交付
- 自测不等于 QA

---

## 三、文章最核心的观念：把“生成”和“评估”拆开

作者先在前端设计任务里验证这个思路，再迁移到长时应用开发。

在前端设计实验里，模型不再只负责“做页面”，而是进入一个闭环：

- `generator` 负责生成
- `evaluator` 负责评分和批评
- 评审反馈回流给生成者
- 生成者继续迭代

这里真正关键的不是多 Agent 本身，而是：

**一旦评估者独立存在，系统里就出现了真正的负反馈回路。**

没有这个回路，系统天然会朝“差不多就行”滑落。  
有了这个回路，系统才有机会朝“更严格、更完整、更有辨识度”推进。

---

## 四、主观问题为什么也能被工程化

前端设计部分有一个很强的启发：  
审美不是完全不可操作的黑箱，关键在于先把“好设计”改写成可评分标准。

作者把前端设计拆成四个维度：

- 设计整体性
- 原创性
- 工艺质量
- 功能可用性

其中最值得记下的是作者的权重取舍：

- 默认模型往往已经能把“工艺”和“可用性”做到及格
- 真正缺的是“整体性”和“原创性”
- 所以评估标准必须显式惩罚模板化、默认组件味、典型 AI 审美套路

这说明一个重要判断：

**评价标准不仅是验收工具，同时也是生成方向的 steering 工具。**

标准怎么写，最后的产物就会往哪里收敛。

---

## 五、迁移到长时应用开发后的结构

在长时 coding 任务里，作者把系统扩成三个角色：

- `planner`
- `generator`
- `evaluator`

它们各自解决不同问题。

### 1. planner 解决“需求过短，系统容易低配实现”

用户往往只给一句话需求。  
如果直接让模型开始写，很容易出现：

- 功能范围收得太窄
- 先做眼前最容易做的部分
- 缺少产品层面的完整性

所以 planner 先把短 prompt 扩成完整 spec。  
文章里特别强调，planner 更偏产品和高层技术方向，而不是先把细节实现写死。

这背后的判断很合理：

**过早写死技术细节，会把一开始的错误放大到整个实现过程。**

### 2. generator 解决“长任务需要持续推进”

generator 负责真正实现。  
在旧版 harness 中，它按 sprint / feature 增量推进；在更新后的版本里，随着模型能力提升，作者开始尝试减少这种人工拆分。

这里能看出作者并不迷信固定流程。  
他不是先假设“永远需要 sprint 化”，而是把 sprint 看成一种阶段性补丁。

### 3. evaluator 解决“看起来像成品，但实际并不可用”

这部分是全文最工程化、也最有现实感的地方。

作者观察到，应用常常会出现一种假象：

- 页面看起来挺完整
- 功能入口也都有
- 但真正点击、交互、跑通一遍，就会暴露出真实 bug

所以 evaluator 不是看代码评论，也不是只跑静态检查，而是直接使用 Playwright 像用户一样测试：

- 点 UI
- 调 API
- 看数据库状态
- 按验收标准判定是否达标

这篇文章的一个强观点就是：

**对长时应用开发来说，“可运行”和“可交付”之间隔着一个真实使用层。**

如果没有这个验证层，Agent 很容易把“像样子”误判成“已经完成”。

---

## 六、一个有趣设计：sprint contract

在旧版三 Agent harness 中，每个 sprint 开始前，generator 和 evaluator 会先协商一份 `sprint contract`：

- 这轮要交付什么
- 什么叫完成
- 如何验证完成

这个设计非常关键：

- 用户故事通常太高层
- 具体代码实现又太底层
- 通过这一步骤来弥合用户故事和可测试实现之间的差距

如果没有中间这一层，开发过程很容易出现“做了一堆东西，但不是要做的那件事”。

所以这份 contract 的作用不是文档化，而是：

**把高层需求翻译成当前轮次可测试、可否决、可交付的局部目标。**

这是文章里最像真实工程管理的地方。

---

## 七、这篇文章对“上下文”最有价值的判断

文章没有把上下文问题只理解成“token 不够”，而是更进一步指出：

- 上下文一长，模型会失去工作稳定性
- 压缩历史不总是够用
- 对某些模型来说，彻底 `reset` 再做结构化交接会更有效

这里我认为最该记住的是：

**上下文管理不是节省 token 的优化，而是维持任务连贯性的主机制。**

这和很多简单理解不同。  
很多人把 compaction、reset、handoff 当成外围技巧；本文实际上把它们提升到了 harness 主体设计的位置。

---

## 八、文章真正想表达的不是“多 Agent 更好”

如果只看表面，很容易把这篇文章读成：

- 用多个 Agent
- 用更复杂的 harness
- 跑更久
- 花更多钱

但作者后半段其实在强调另一件事：

**harness 的每一个组件，都只是对“模型当前还做不到什么”的临时补偿。**

这也是为什么当 Opus 4.6 能力提升后，作者马上开始删结构：

- 去掉某些 context reset 依赖
- 尝试移除 sprint 构造
- 把 per-sprint 评审改成更晚阶段的 QA

这里最成熟的地方在于，作者没有把原来有效的 harness 神圣化。  
他把 harness 看成一组随模型能力变化而不断迁移的承重件。

所以更准确的说法不是：

“复杂 harness 会越来越重要”

而是：

**有价值的 harness，永远是在重新定位复杂度，而不是一味累加复杂度。**

---

## 九、文章给我的几个稳定结论

### 1. 长时 Agent 的问题，本质上越来越像组织设计问题

真正影响结果的，已经不只是模型会不会写，而是：

- 角色怎么分
- 信息怎么流
- 谁来否决
- 哪些状态要外置
- 什么时候交接

这更像在设计一个小型软件团队，而不是一个单点“超级 prompt”。

### 2. 评审标准本身就是生成控制面

无论是前端设计还是应用开发，标准写法会直接改变系统行为。  
你给出什么维度、什么阈值、什么反例，系统就会往那个方向训练式收敛。

### 3. “看起来像完成”是长任务里非常危险的假象

Agent 做应用时最容易出现的不是完全不会做，而是做出一个有界面、有入口、有些流程，但关键路径仍然断裂的东西。  
文章里的 evaluator 正是在对抗这种假象。

### 4. harness 必须随着模型升级主动重审

旧补丁会过时。  
继续保留它们，不一定提升质量，反而会增加成本、延迟和系统复杂度。

---

## 十、简短评价

这篇文章最有价值的地方，是把 Agent engineering 往前推了一步：

- 不是只谈 prompt
- 不是只谈工具
- 不是只谈 context
- 而是把它们放进一个完整的长时任务编排框架里看

如果只保留一句自己的批注，我会记这句：

**长时程 Agent 的核心难点，不是让模型持续输出，而是让系统持续保持“知道自己在做什么、还差什么、谁来判定够不够”。**
