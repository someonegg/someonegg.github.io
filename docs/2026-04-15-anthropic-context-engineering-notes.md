这三篇文章可以合并为一条主线：

1. `上下文工程（context engineering）` 是从“写提示词”升级到“管理整轮推理可见信息”的工程问题。

## 1. 从提示词工程（Prompt Engineering）到上下文工程（Context Engineering）

Anthropic 的核心定义是：
`提示词工程（prompt engineering）` 处理“提示词怎么写”，`上下文工程（context engineering）` 处理“哪些 token 在当前轮应该进入窗口”。

这意味着焦点从静态文案转向动态编排：系统提示、工具定义、文档、记忆、消息历史、工具结果都需要被持续筛选。

![Prompt vs Context](./assets/anthropic-context-engineering/01-prompt-vs-context.webp)

### 1.1 为什么这件事突然变重要

文章强调：随着智能体跨多轮、长时程执行，信息会持续累积；上下文窗口虽然增大，但注意力预算依然有限。上下文不是“越多越好”，而是“高信号优先”。

这和“上下文腐烂（context rot）”问题一致：当 token 越来越多，模型对关键事实的召回与精确推理会下降。

```text
为什么上下文工程对构建功能强大的智能体至关重要？

尽管语言学习模型（LLM）速度很快，而且能够处理越来越大的数据量，但我们观察到，它们和人类一样，在某个阶段也会失去焦点或感到困惑
。对“大海捞针”式基准测试的研究揭示了“上下文腐烂”的概念：随着上下文窗口中词元数量的增加，模型准确回忆上下文信息的能力会下降。

虽然有些模型的性能退化程度比其他模型更缓和，但所有模型都存在这一特性。因此，上下文必须被视为一种边际收益递减的有限资源。与工作记
忆容量有限的人类类似，LLM（逻辑学习模型）也存在“注意力预算”，用于解析大量上下文信息。每引入一个新的词元，都会消耗一部分注意
力预算，这就增加了 LLM 需要精心筛选可用词元的必要性。

这种注意力稀缺性源于 LLM 的架构限制。LLM 基于 Transformer 架构，该架构允许每个词元关注整个上下文中的所有其
他词元。这导致 n 个词元之间存在 n² 个成对关系。

随着上下文长度的增加，模型捕捉这些成对关系的能力会逐渐捉襟见肘，从而在上下文大小和注意力集度之间产生一种天然的矛盾。此外，模型的
注意力模式是从训练数据分布中发展而来的，而训练数据分布中短序列通常比长序列更为常见。这意味着模型在处理上下文范围内的依赖关系方面
经验较少，并且缺乏专门的参数。

位置编码插值等技术允许模型通过将其适应最初训练的较小上下文来处理更长的序列，尽管这会降低模型对词元位置的理解能力。这些因素造成了
性能的梯度变化，而非断崖式下降：模型在较长的上下文中仍然保持着很高的能力，但与在较短上下文中的表现相比，其信息检索和长程推理的精
度可能会降低。

这些现实意味着，周密的上下文工程对于构建有能力的智能体至关重要。
```

## 2. 有效上下文的第一原则：清晰、适量、可演化

### 2.1 系统提示的校准区间

Anthropic 给出的图很直观：

- 过窄：把流程写成脆弱的硬编码脚本，维护成本高，泛化差。
- 过宽：只有口号式方向，缺少可执行约束。
- 合适：职责、边界、决策框架足够清楚，但保留任务内的推理弹性。

![Calibrating system prompt](./assets/anthropic-context-engineering/02-calibrating-system-prompt.webp)

文中建议把系统提示结构化分段（如背景、指令、工具指导、输出要求），本质是让“约束可读、可检索、可修改”。

### 2.2 预注入与运行时检索的平衡

`effective-context-engineering` 文中强调了一个常被忽略的权衡：

- 预注入多：首轮快，但容易带入过期或低相关信息。
- 运行时检索多：更贴近当前任务，但速度慢。

它推荐的是混合策略，不是极端策略。

```text
在《为人工智能代理编写工具——与人工智能代理一起》一文中，我们讨论了如何构建易于语言学习者理解且功能重叠最小的工具。与精心设计的
代码库类似，工具也应具备自包含性、容错性强，并且其预期用途清晰明确。同样，输入参数也应具有描述性、无歧义，并能充分发挥模型的固有
优势。

我们最常见的故障模式之一是工具集过于臃肿，涵盖的功能过多，或者导致在选择工具时出现模糊不清的决策点。如果人类工程师都无法明确判断
在特定情况下应该使用哪个工具，就不能指望人工智能代理做得更好。正如我们稍后将讨论的，为代理精心挑选一套最小可行工具集，也有助于在
长期交互过程中更可靠地维护和精简上下文信息。

提供示例，也称为少样本提示，是一种众所周知的最佳实践，我们一直强烈建议这样做。然而，团队常常会在提示中塞入一大堆极端情况，试图阐
明 LLM 在特定任务中应遵循的所有规则。我们不建议这样做。相反，我们建议努力精心挑选一组多样化的、规范的示例，以有效地展现智能
体的预期行为。对于 LLM 而言，示例胜过千言万语。

我们针对上下文各个组成部分（系统提示、工具、示例、消息历史记录等）的总体指导原则是：要深思熟虑，保持上下文信息丰富且简洁明了。现
在，让我们深入探讨如何在运行时动态检索上下文。
```

```text
让智能体自主地导航和检索数据，还能实现渐进式信息披露——换句话说，智能体可以通过探索逐步发现相关的上下文。每次交互都会产生上下文
信息，为下一步决策提供依据：文件大小暗示复杂性；命名规则暗示用途；时间戳可以作为相关性的参考。智能体可以逐层构建理解，仅在工作内
存中保留必要信息，并利用笔记策略来增强信息的持久性。这种自我管理的上下文窗口使智能体能够专注于相关的信息子集，而不是被冗长但可能
无关的信息淹没。

当然，这其中存在权衡：运行时探索比检索预先计算的数据要慢。不仅如此，还需要精心设计和周密规划，以确保逻辑逻辑模型（LLM）拥有合
适的工具和启发式方法，从而有效地驾驭其信息环境。如果没有适当的指导，智能体可能会因为误用工具、陷入死胡同或未能识别关键信息而浪费
上下文信息。

在某些情况下，最有效的智能体可能会采用混合策略：预先检索一些数据以提高速度，然后根据自身判断进行进一步的自主探索。自主程度的“合
适”界限取决于具体任务。Claude Code 就是一个采用这种混合模型的智能体：CLAUDE.md 文件会被预先简单地放入上下
文中，而像 glob 和 grep 这样的基本工具则允许它导航环境并即时检索文件，从而有效地绕过了过时的索引和复杂的语法树等问题
。

混合策略可能更适合内容动态性较低的场景，例如法律或金融领域。随着模型能力的提升，智能体设计将朝着让智能模型自主行动的方向发展，并
逐步减少人工干预。鉴于该领域发展日新月异，“做最简单有效的方法”可能仍将是我们对基于 Claude 构建智能体的团队的最佳建议。
```

## 3. 长时任务中的失效模式与执行框架（Harness）思路

`effective-harnesses-for-long-running-agents` 把问题拆得很工程化：

- 每次新会话都像“新同事接班”，如果没有交接工件，必然反复踩坑。
- 仅靠上下文压缩（compaction）不够，还需要稳定的“环境管理+进度管理+验证纪律”。

### 3.1 图中表格的文本化整理

| 问题 | 初始化智能体 | 编码智能体 |
|---|---|---|
| 过早宣布完成 | 建立结构化 feature list（默认未通过） | 会话开头读取 feature list，只做一个 feature |
| 环境遗留缺陷或进度不透明 | 初始化 git 与进度文件 | 开始先读进度和日志，结束写提交与进度 |
| 功能过早标记完成 | 建立可验证的功能清单 | 先自验，再标记 passing |
| 每轮都在摸索启动方式 | 提供 `init.sh` | 每轮先读取并按脚本启动 |

这张表的价值不在“提示词措辞”，而在“会话协议”。

## 4. 三种核心上下文策略（应对窗口限制）

Anthropic 在 context 文章里给出的三条主策略，分别解决不同类型的长任务连续性问题。

### 4.1 压缩续航（Compaction）

对接近窗口上限的历史做高保真摘要，重启窗口继续执行。关键点是：

1. 先保召回，再做精简。
2. 优先移除冗余工具结果这类“低风险噪声”。

```text
压缩（Compaction）

压缩是指：当对话接近上下文窗口上限时，先对内容做摘要，再用该摘要重新开启一个新的上下文窗口。压缩通常是上下文工程中的第一根杠杆，
用于提升长时程任务的一致性。其核心目标是高保真提炼窗口内容，让智能体在性能尽量不下降的前提下继续工作。

以 Claude Code 为例，我们会把消息历史交给模型做总结与压缩，保留最关键细节。模型会保留架构决策、未解决缺陷和实现细节
，同时丢弃冗余的工具输出或消息。随后，智能体在这份压缩上下文基础上继续执行，并附带最近访问的文件，从而在不受窗口限制困扰的情况下
保持连续性。

压缩的关键在于“保留什么、丢弃什么”的取舍。过度压缩会丢失细微但关键的信息，而这类信息的重要性往往在后续阶段才显现。对实现压缩系
统的工程师而言，建议在复杂智能体轨迹上仔细调试压缩提示词：先优先提高召回，确保相关信息不漏，再迭代提升精度，逐步去掉冗余内容。

一个常见且低风险的冗余内容是历史工具调用结果：某个工具在很早之前已经调用过，智能体通常不必反复看到原始返回。最安全、最轻量的压缩
方式之一，就是清理工具结果；这也已作为 Claude Developer Platform 的功能发布。
```

### 4.2 结构化笔记（Structured Note-taking）

将任务状态写入上下文外的结构化笔记，按需回拉。它解决的是“会话断点后状态蒸发”。

```text
结构化笔记（Structured note-taking）

结构化笔记，也可称为智能体记忆（agentic memory），是一种让智能体定期把笔记写到上下文窗口之外并持久化保存的技术。这
些笔记会在后续需要时被重新拉回到上下文窗口。

这种策略以很小的开销提供了持久记忆。类似 Claude Code 维护待办清单，或你的自定义智能体维护 `NOTES.md` 文
件，这种简单模式可以让智能体在复杂任务中持续跟踪进度，保住那些原本会在几十次工具调用后丢失的关键上下文和依赖关系。

“Claude 玩宝可梦”的案例展示了记忆在非编程场景中的能力提升：智能体可以在数千步流程中保持精确计数，例如“最近 1234
步里，我一直在 1 号道路训练宝可梦，皮卡丘已从目标 10 级中提升了 8 级”。即便没有被显式要求采用某种记忆结构，它也会逐步
形成已探索区域的地图、记住已解锁的重要里程碑，并记录战斗策略，从而学习不同对手下更有效的攻击方式。

在上下文重置后，智能体可读取自己的笔记，继续多小时的训练序列或地牢探索。这种跨摘要步骤的一致性，使长周期策略成为可能；如果所有信
息都硬塞在 LLM 上下文窗口里，这一点几乎无法实现。

随着 Sonnet 4.5 发布，我们在 Claude Developer Platform 公测了记忆工具，通过文件化系统让窗
口外信息的存储与查询更容易。这使智能体可以长期积累知识库、跨会话维护项目状态，并在不把所有信息放入当前上下文的前提下引用历史工作
。
```

### 4.3 子代理架构（Sub-agent Architectures）

主代理做统筹，子代理在干净上下文里做深挖，再返回蒸馏摘要。它解决的是“单代理上下文过载”。

```text
子代理架构（Sub-agent architectures）

子代理架构为绕开上下文限制提供了另一条路径。与其让一个智能体在整个项目中维护全部状态，不如让专门的子代理在更干净的上下文窗口里处
理聚焦任务。主代理负责高层计划与协调，子代理负责深度技术工作或借助工具检索相关信息。每个子代理都可能进行大规模探索（数万
token 甚至更多），但只返回凝练后的摘要结果（常见为 1000-2000 token）。

这种方式实现了清晰的关注点分离：细粒度检索上下文被隔离在子代理内部，主代理则聚焦于结果综合与分析。该模式在《How we
built our multi-agent research system》中有讨论，在复杂研究任务上相较单代理系统有明显提升。

三种方式的选择取决于任务特性。例如：

• 压缩（Compaction）适合大量来回交互、需要保持对话连续性的任务；
• 结构化笔记（Note-taking）适合里程碑清晰的迭代开发；
• 多智能体架构（Multi-agent）适合复杂研究与分析，且并行探索收益高。

即便模型持续进步，如何在长交互中保持一致性，仍会是构建高效智能体的核心难题。
```

### 4.4 三策略对照

| 策略 | 优势 | 代价 | 更适用任务形态 |
|---|---|---|---|
| 压缩（Compaction） | 保持会话连续 | 摘要质量决定上限 | 多轮往返、上下文连续对话 |
| 结构化笔记（Structured note-taking） | 跨会话状态稳定 | 需要笔记结构纪律 | 迭代式任务、里程碑明确 |
| 子代理（Sub-agents） | 并行探索、上下文隔离 | 编排复杂度更高 | 复杂研究、分工明确的大任务 |

## 5. 什么是“好工具”：从 API 暴露到 Agent 认知设计

`writing-tools-for-agents` 的核心观点是：
智能体工具不是“把已有 API 原样包一层”，而是“按智能体的认知与上下文成本重新设计接口”。

```text
什么是工具？

在计算机领域，确定性系统在给定相同输入的情况下每次都会产生相同的输出，而非确定性系统（如智能体）即使在相同的初始条件下也可能产生
不同的响应。

传统软件开发中，我们实际上是在确定性系统之间建立一种契约。例如，像这样的函数调用 getWeather("NYC") 每次都会以
完全相同的方式获取纽约市的天气。

工具是一种新型软件，它体现了确定性系统与非确定性主体之间的一种契约。当用户询问“我今天需要带伞吗？”时，主体可能会调用天气工具，
根据常识回答，甚至可能先询问一些关于位置的澄清问题。有时，主体可能会出现幻觉，甚至无法理解如何使用工具。

这意味着我们需要从根本上重新思考我们为代理编写软件的方法：我们不再像为其他开发人员或系统编写函数和 API 那样编写工具和
MCP 服务器，而是需要为代理设计它们。

我们的目标是利用各种工具来探索多种有效的策略，从而扩大智能体能够有效解决各种任务的范围。幸运的是，根据我们的经验，那些对智能体来
说最“符合人体工程学”的工具，最终也出乎意料地容易被人类理解和接受。
```

### 5.1 工具不是越多越好

Anthropic 明确指出：过多或重叠工具会增加选择歧义，反而降低成功率。

```text
为智能体选择合适工具（Choosing the right tools for agents）

工具并非越多越好。我们常见的错误是：只是把现有软件功能或 API 端点再包一层，而不判断这些工具是否真的适配智能体。原因在于，智
能体相较传统软件具有不同的“可供性（affordances）”，即它理解和选择可行动作的方式并不相同。

LLM 智能体的“上下文”是受限资源（一次可处理信息量有限），而计算机内存通常廉价且充足。以“在通讯录中找联系人”为例，传统软件
可以逐条高效扫描列表。

但如果 LLM 智能体调用一个返回“全部联系人”的工具，再逐 token 从头读到尾，就会把稀缺上下文浪费在不相关信息上（类似你
翻通讯录每一页硬搜）。更自然、更高效的方式，是先跳到相关范围（例如按字母定位）。

因此我们建议：先围绕高价值工作流设计少量高质量工具，并让其对齐你的评测任务，再逐步扩展。以通讯录为例，更应优先实现
`search_contacts` 或 `message_contact`，而不是 `list_contacts`。

工具还可以在内部整合多个离散操作（或 API 调用）。例如，工具可在返回中附带相关元数据，或把常见链式多步流程折叠为一次工具调用
。
```

这也是为什么它建议优先做少量高价值、任务导向工具，而非细碎底层接口全集。

### 5.2 工具描述本身就是提示工程

工具名、参数名、输入输出约束会直接影响调用质量。文章示例里强调参数语义应去歧义（例如使用更具体的 ID 命名），并通过评测来验证改动收益。

```text
用提示工程优化工具描述（Prompt-engineering your tool descriptions）

提升工具效果最有效的方法之一，是对工具描述与规格本身做提示工程。因为这些内容会进入智能体上下文，它们会共同引导智能体形成更有效的
工具调用行为。

编写工具描述和规格时，可以把自己想象为在给团队新人讲解工具。你默认掌握但未写明的背景信息（例如专用查询格式、术语定义、底层资源关
系）都应显式写出。要通过清晰描述并配合严格数据模型，避免输入输出歧义。特别是参数命名必须明确：例如不要用 `user`，而应使用
`user_id`。

借助评测，你可以更有把握地衡量这些提示工程改动的影响。工具描述即使只做小幅优化，也可能带来显著改进。Anthropic 在精修工
具描述后，Claude Sonnet 3.5 在 SWE-bench Verified 上取得了领先表现，错误率显著下降，任务完
成率明显提升。

更多工具定义最佳实践可参考 Developer Guide。若你在为 Claude 构建工具，也建议了解工具如何动态加载到
Claude 的系统提示中。若你在编写 MCP 服务器工具，工具注解可明确哪些工具需要开放世界访问、哪些工具可能触发破坏性变更。
```

2. 需要“上下文治理 + 会话协议 + 工具契约”协同设计。
3. 评测应贯穿全过程，否则容易把局部成功误判为系统能力。

## 7. 阅读后的关键判断（笔记结论）

1. `上下文工程（context engineering）` 的本质是信息预算管理，而不是提示词修辞。
2. 智能体工程从“单轮表现”进入“多轮稳定性”阶段，成功标准必须包含连续性与可恢复性。
3. 工具设计已经成为与模型能力同等重要的杠杆：边界清晰、语义明确、返回高信号、评测驱动。
4. 在模型持续变强的背景下，最佳实践不是堆更多规则，而是保持系统尽量简单并可迭代演化。

## 8. 补充：Reasoning Shift（上下文如何悄悄缩短大模型的推理）

“上下文腐烂（context rot）”解释：

```text
第一层归咎于检索失败。2023 年斯坦福在《Lost in the Middle》里指出，模型在长文本中形成 U 形注意力曲线，中间区域被
忽视。行业的应对是 RAG，把长文本切碎，用向量检索喂最相关的片段。

第二层推翻了第一层。2025 年的论文《Context Length Alone Hurts LLM Performance Despite Perfect Retrieval》做了
实验：把所有无关内容遮掉，强迫模型只看需要的信息，性能还是滑坡 13.9% 到 85%。即使把无关内容全部替换成空白符，结果依然
如此。问题不在找不到信息，而是上下文的纯长度本身就在伤害推理。行业的应对是 Context Engineering（上下文工程）。压缩
上下文、管理窗口、浓缩历史，死死压住 Token 数量。

第三层来自 Microsoft 和 Salesforce 的联合研究（2025 ICLR）。他们发现把完整指令切成多轮喂给模型，跨六个任务和十五个
模型，平均性能暴跌 39%。只要某一轮走错一步，后面就彻底迷失。行业在 Harness 里搭起了最核心的重型防御：交班管控、定期强制
验证中间结果、以代码仓库为唯一事实来源，绝不允许模型靠自己去记上一轮发生了什么。

三层问题，三层脚手架。但这些都只是现象层的发现。回看第二层，研究者发现长度本身就有害，跟信息质量毫无关系。行业能做的只有
物理管控长度。但如果问题的根源根本不在长度本身呢？
```

这篇文章的主观点不是简单重复“上下文越长越差”，而是提出一个更强的解释：
**模型在长上下文里不一定是“看不见信息”，更可能是“主动减少思考投入”，尤其减少答案后的自检与反思。**

它的论证链大致是四步：

1. 先承认行业已有解释与对应工程补丁：检索增强、上下文压缩、harness 协议。  
2. 再强调最新论文现象：即使信息可得，推理 token 仍显著收缩，并随上下文复杂度上升而加剧。  
3. 关键转折是“收缩位置”：模型往往不是更慢找到候选答案，而是更早停止复核，因此更像认知投入下降，而非纯检索障碍。  
4. 最后把该现象连接到 Anthropic 的内部状态研究，提出一个可检验假设：  
   长上下文下的“偷懒式收工”与 reward hacking 可能共享部分内部驱动机制，因而未来可通过训练信号与推理时干预去调控。

按这条观点链，文章给出的工程含义是：

1. `Harness Engineering` 仍有现实必要性，但其角色更像“外部稳定器”，不是终局解法。  
2. 只在外部加脚手架会遇到边际收益递减，应同步建设“内部状态可观测 + 训练侧约束 + 推理时行为干预”。  
3. 对生产团队更务实的路径是“双轨制”：  
   短期靠 harness 保交付，长期把投入转向可减少认知收缩的模型与训练机制。
