# 多 Hook 协作逻辑

## 核心关系

多个 Hook 会形成链式结构：

```text
hook_1(current) -> hook_2 -> ... -> origin
```

其中 `current` 表示当前入口（通常是系统可见的链头）。

## 卸载时的协作策略

协作式实现可按下面逻辑处理：

1. 先判断 `current` 是否是自己
2. 如果是自己，说明可以安全卸载
3. 如果不是自己，不立即强拆，而是设置“延迟卸载”标志
4. 当本 Hook 过程后续再次收到消息时，再尝试卸载（回到步骤 1）

这套逻辑的关键是：不破坏当前链头关系，等到自己处于可安全卸载位置再执行卸载。

## 当其他 Hook 不协作时

其他模块如果采用“更暴力”的方式，常见有三类：

1. 不还原链，直接卸载模块
2. 强行把入口还原到自己后一个节点，再卸载
3. 直接还原到宿主（如 explorer）最初始状态，再卸载

## 可能结果（对应原始语义）

针对上面三类做法，常见后果是：

1. 选择 3：通常不会让 explorer 崩溃，但会导致我们收不到消息，从而无法完成自己的卸载流程
2. 选择 2：
   - 若对方在我们后面，效果接近“我们无法被正常卸载”
   - 若对方在我们前面，则我们可能还能正常卸载
3. 选择 1：最容易引发崩溃

## 结论

多 Hook 场景中，只要各方采用同一套“可安全时卸载、不可安全时延迟”的协作逻辑，链可保持稳定。

一旦混入不协作实现，行为会迅速复杂化，且崩溃风险显著上升。
